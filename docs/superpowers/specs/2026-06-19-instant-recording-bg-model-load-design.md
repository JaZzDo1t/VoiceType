# Дизайн: мгновенный захват записи + фоновая загрузка модели

**Дата:** 2026-06-19
**Статус:** утверждён (brainstorming)
**Затрагивает:** `voicetype/src/core/recording_session.py`, `voicetype/src/core/model_loader.py`, `voicetype/src/app.py`

## Проблема и цель

Когда модель выгружена (по таймауту бездействия, `unload_timeout` по умолчанию 10с), нажатие хоткея запускает `_reload_models_then_start()`: модель грузится в главном потоке (1–3с, блокирующе) и **только после** этого вызывается `_do_start_recording()` — стартует микрофон и появляется красный индикатор. Из-за этого первые слова теряются: пользователь ждёт загрузку, прежде чем запись реально начнётся.

**Цель:** при нажатии хоткея захват аудио (и красный индикатор) начинается немедленно; модель догружается в фоне; накопленный звук распознаётся, как только модель готова. Ноль ожидания в повседневном случае (перезагрузка после авто-выгрузки).

## Ключевые факты из кода (делают решение реальным)

- `AudioCapture._audio_queue` — `queue.Queue()` **без ограничения размера** (`audio_capture.py:46`). Звук, захваченный во время загрузки, накапливается и не теряется.
- Захват PyAudio работает в собственном потоке через `stream_callback` — не зависит от модели и главного потока.
- **Ограничение:** модель Whisper (CTranslate2) обязана грузиться в **главном потоке** при работающем Qt (иначе краш — см. CLAUDE.md). Поэтому поток распознавания НЕ должен сам подгружать модель; загрузку инициирует главный поток, а поток распознавания ждёт сигнала готовности.

## Принятые решения

- **Подход:** мгновенный захват + фоновая загрузка (сохраняет авто-выгрузку и экономию VRAM). Отклонены варианты «не выгружать вовсе» и «гибрид с большим таймаутом».
- **Охват:** случай авто-выгрузки (повседневный) + холодный старт. Для холодного старта — перестановка порядка запуска (хоткеи раньше предзагрузки) + снятие раннего выхода `start()` по `_models_loaded`.
- **Осознанное ограничение (вне scope):** если нажать хоткей ровно во время активной первой загрузки модели при старте приложения (1–3с заморозки главного потока), нажатие встанет в очередь и захват начнётся после загрузки — узкое окно. Полное покрытие требует «пред-захвата» из потока pynput (обход главного потока) — отложено как опциональное расширение, в этой работе НЕ делается.
- Изменения только в тайминге/порядке. Логика VAD/Whisper/транскрипции не трогается.

## Архитектура

Распараллеливаем две независимые вещи, которые сейчас последовательны: (1) захват аудио и (2) загрузку модели. Связь между ними — `threading.Event` `_model_ready` в `RecordingSession`. Захват и поток распознавания запускаются сразу; поток распознавания блокируется на `_model_ready` и сливает накопленную очередь, как только событие выставлено главным потоком после успешной загрузки.

## Компоненты

### `recording_session.py` — `RecordingSession`

Новое поле: `self._model_ready = threading.Event()`.

**`start()`** (переписан):
```text
if _is_recording.is_set(): return
если _app._recognizer отсутствует → _app._models._create_whisper_recognizer() (ленивое создание)
_model_ready.clear()
_do_start_recording()                      # захват + поток распознавания + красный индикатор, СРАЗУ
если _app._recognizer.is_loaded():
    _model_ready.set()                     # распознаём в реальном времени
иначе:
    _app._models._load_then_signal_ready() # фоновая загрузка в главном потоке
```
Порядок важен: `_do_start_recording()` внутри уже зовёт `set_processing(True)` (блокирует авто-выгрузку и отменяет таймер) — это снимает гонку с авто-выгрузкой до проверки `is_loaded()`. Поток распознавания стартует, но ждёт `_model_ready`, поэтому звать `process_audio` до готовности не будет.

**`_recognition_loop()`** (переписан): сначала ждёт `_model_ready` (с таймаутом 0.1с, проверяя `_is_recording`), затем — текущий цикл слива очереди. Если запись остановили во время ожидания — выходит.

**`signal_model_ready()`** (новый): `self._model_ready.set()`. Вызывается `ModelLoader` после успешной загрузки.

**`abort_load_failure(detail)`** (новый): аккуратно гасит запись при провале фоновой загрузки — `_is_recording.clear()`, остановка таймера уровня и `AudioCapture`, `recognizer.set_processing(False)`, сброс буфера, `_error_signal`/уведомление, возврат трея в READY. Буфер записи отбрасывается.

**`stop()`**: по сути без изменений; `get_final_result()` уже безопасен при незагруженной модели (вернёт None). `_model_ready` повторно очищается в `start()` следующей сессии.

### `model_loader.py` — `ModelLoader`

Новый путь «загрузить в фоне → подать сигнал», параллельный записи (вместо «загрузить → потом стартовать»):

- **`_load_then_signal_ready()`**: показать статус загрузки в test-tab; `QTimer.singleShot(50, _do_load_then_signal_ready)` (загрузка в главном потоке).
- **`_do_load_then_signal_ready()`**: диагностика окружения (`_check_environment`); при провале → `_app._recording.abort_load_failure(...)`. Иначе `set_processing(True)`, `cancel_unload_timer()`, `load_model()`. При успехе: эмит `_whisper_status_signal`/`_vad_status_signal`, `_app._models_loaded = True`, и **если запись ещё идёт** → `_app._recording.signal_model_ready()`, **иначе** `set_processing(False)` (запись успели остановить — разрешаем авто-выгрузку). При провале → `abort_load_failure(...)`.

Старые `_reload_models_then_start` / `_do_reload_models_then_start` удаляются (заменяются новым путём).

**Защита от двойной загрузки при холодном старте:** в начале `_do_load_models` (стартовая предзагрузка) — `if _app._recording.is_recording(): return` (запись уже сама грузит модель; предзагрузка не должна делать `_unload_models()` и обнулять активный recognizer).

### `app.py` — `VoiceTypeApp`

- **`RecordingSession.start()` сам решает путь** — ранний выход по `not _models_loaded` снимается из `start()` (логика переносится: запись допустима до готовности, recognizer создаётся лениво).
- **Перестановка запуска в `initialize()`:** сначала поднять hotkey listener, затем — отдельным `QTimer.singleShot` с небольшой задержкой — предзагрузку (`_models._load_models_async()`), чтобы между поднятием хоткеев и активной загрузкой было окно, в котором нажатие ловит мгновенный захват. (Текущий порядок — предзагрузка `singleShot(50)` раньше хоткеев `singleShot(100)` — инвертируется.)

## Поток данных

Хоткей (модель выгружена): `_on_hotkey_triggered` → `toggle_recording` → `RecordingSession.start()` → `_do_start_recording()` (микрофон+поток+красный индикатор, мгновенно) → `_load_then_signal_ready()` (главный поток, параллельно). Поток распознавания ждёт `_model_ready`, звук копится в очереди. Загрузка завершилась → `signal_model_ready()` → поток сливает накопленную очередь (VAD быстрый, Whisper догоняет) и далее работает в реальном времени.

## Обработка ошибок

- Провал загрузки/диагностики во время записи → `abort_load_failure()`: запись гасится, буфер отброшен, показана ошибка, трей возвращён в READY, авто-выгрузка снова разрешена.
- `stop()` во время ожидания загрузки: главный поток сериализован Qt — `stop()` и `_do_load_then_signal_ready` не пересекаются. Если запись остановлена до завершения загрузки, после загрузки вызывается `set_processing(False)` (не `signal_model_ready`), чтобы не залипла блокировка авто-выгрузки.

## Память

Очередь во время загрузки: 16кГц·int16·моно ≈ 32 КБ/с. 3с ≈ 96 КБ, даже 60с ≈ 2 МБ — пренебрежимо.

## Верификация

- **Синтаксис/импорт:** `recording_session.py`, `model_loader.py`, `app.py` импортируются без ошибок.
- **Существующие тесты** (whisper/diagnostics) остаются зелёными (страховка соседнего кода).
- **Ручная проверка запуском** (основная гарантия; автотесты на Qt+потоки вне scope):
  1. Дождаться авто-выгрузки по таймауту → хоткей → **красный индикатор и захват сразу**; надиктовать фразу с первого слова → текст распознан целиком (начало не потеряно).
  2. Обычная запись при загруженной модели → работает как раньше (real-time партиал/финал).
  3. Стоп во время записи → финальный результат вставлен.
  4. Смена модели/языка в настройках → reload отрабатывает.
  5. Провал загрузки (например, недоступна CUDA) во время записи → запись гаснет, показана ошибка, трей в READY, приложение не висит.

## Критерий готовности

При нажатии хоткея с выгруженной моделью захват и красный индикатор появляются немедленно, начало фразы не теряется; модель догружается в фоне; накопленный звук распознаётся; ошибки загрузки гасят запись корректно; обычная запись и смена модели работают как прежде.
